Intelligent Cities APIs

Intelligent Cities API Overview

Each Intelligent Cities API provides asset and location information, with additional URI resources and data collected from sensors installed on intelligent lighting fixtures in roadways and parking areas. Each API service is described below.

Note:

Please contact your Current, Powered by GE representative to obtain the following before you start developing:

  1. client-id and client-secret to generate client tokens
  2. predix-zone-id for each service
  3. services' URLs.

Parking Planning

The Parking Planning service provides parking metadata that is collected from intelligent lighting sensors installed along public roadways or in parking lots. Each time a vehicle enters or leaves a parking space, a timestamped event is created and stored in the Predix cloud.

This data can be obtained from a single sensor or a group of sensors, as needed, and can provide valuable insights about the parking activity in a given region. The data can also be used to improve safety and parking convenience.

Pedestrian Planning

The Pedestrian Planning service provides pedestrian metadata that is collected from intelligent lighting sensors installed along public roadways or in parking lots.

As pedestrians enter or leave a region of interest, the service creates and stores a timestamped event, counting each event. This data can be obtained from a single sensor or a group of sensors, as needed, to provide insights to city officials and local businesses about pedestrian usage in a given region. The data can also be used to enhance safety and convenience.

Traffic Planning

The Traffic Planning service provides vehicle metadata collected from intelligent lighting sensors installed along public roadways.

As vehicles pass by these sensors, information such as speed, direction, lane use, and volume (counts) are collected and stored with a time-stamp in the Predix cloud. The data can be viewed from any number of intelligent street lights in a given region and can present both historical and near real-time information.

The density, range, and flexibility of this sensor data can provide insights into traffic patterns at different intervals. Along with facilitating the planning for future safety and convenience enhancements, the data can reveal opportunities to optimize traffic flows.

One response is provided for each car that traverses the area monitored by the device.

Situational Awareness (Media)

The Situational Awareness service provides access to images, video, and other media collected from intelligent lighting sensors installed along public roadways or parking lots.

The media can be requested using a date range starting near real-time and ending based on availability. This data can be obtained from a single sensor or a group of sensors, as needed, to provide insights to city officials that enhance awareness. This service can be used in concert with other Intelligent Environments services to enhance a use case of interest.

Images are captured for a variety of scenarios for vehicles and pedestrians.

WebSocket Information for Intelligent Cities APIs

WebSocket functions with the Intelligent Cities API services, and provides the ability to retrieve near real-time data to obtain insight into events collected by each service.

Parking Planning

There are two ways to retrieve Parking Planning data.
  1. Via WebSocket to retrieve near-real-time data based on:
    • AssetUid
    • Locations
    • LocationUid
  2. Via REST APIs to retrieve historical data based on:
    • AssetUid
    • Locations
    • LocationUid
    Note: You can only retrieve one event type - either PKIN or PKOUT - when retrieving historical data.

The WebSocket URL provides a link to near-real-time data for vehicle parking at the specified monitored location. You must use a WebSocket client to retrieve this data.

Situational Awareness (Media)

The Situational Awareness API returns a link from which you can view a video or image captured by an asset (camera).

There are two ways to retrieve media on demand
  • By AssetUid for Image or Video
  • By LocationUid for Image or Video
There are two ways to retrieve historical data:
  • By AssetUid for Image or Video
  • By LocationUid for Image or Video
  • Retrieve Most Recent Historical Data for Image or Video

Retrieve Media on Demand by AssetUid for Image or Video

When retrieving media on demand:
  1. Submit the HTTPS request. You will receive an imagePollURL or videoPollURL.
  2. Submit the imagePollURL or videoPollURL.
    • The message “Success” returns if the media file you requested is in the database.
    • If the requested file is not in the database, it will retrieve the file from the node (on-demand).
      Note: The timestamp in the request needs to match, up to the millisecond.
    • The message "InProgress" returns if the media file is not in the database. There is no timeout or error. For now, the recommendation is the application be set to a timeout of five minutes, but the developer has the flexibility to choose whatever timeout period they want.
      Note: Media is retained by the node for seven days. If the requested media is older than seven days and is found in the cloud database, the PollURL returns the requested media.

WebSocket Information for Intelligent Cities APIs

Using WebSocket with the Intelligent Cities services: Parking Planning, Traffic Planning, and Pedestrian Planning API services provides the ability to retrieve near-real-time data by assetUID, locations (bbox), or locationUID to obtain insight into the events collected by each service.

WebSocket Information

To retrieve near real-time events, use the WebSocket protocol instead of HTTPS for streaming data. The streaming endpoint format is: wss://<streaming url>

Note: You need to send the bearer token and Predix-Zone-Id while establishing the socket connection.

By assetUid

WebSocket URL: "wss://<production_url>/events"
Note: Predix Tool Kit offers a websocket client that you can use for websocket calls.
Headers Request Payload
Authorization: Bearer <token from trusted issuer> Predix-Zone-Id: <Predix-Zone-Id> {"assetUid":"<assetUid>","eventTypes":["<eventType1>", "<eventType2>"]}

By locations

Headers Request Payload
Authorization: Bearer <token from trusted issuer> Predix-Zone-Id: <Predix-Zone-Id> {"bbox":"<long,lat>","eventTypes":["<eventType1>", "<eventType2>"]}

By locationsUid

Headers Request Payload
Authorization: Bearer <token from trusted issuer> Predix-Zone-Id: <Predix-Zone-Id> {"locationUid":"<locationUid>","eventTypes":["<eventType1>", "<eventType2>"]}
